# ThreadCraft - A Tailoring Business Management System
## Requirements & Technical Specification Document

**Version:** 1.0  
**Date:** November 2, 2025  
**Project Type:** Home-based Tailoring Business Management Web Application

---

## 1. Executive Summary

A web-based application designed for a home-based tailoring business to manage client information, measurements, orders, and billing. The system will streamline the measurement recording process, order tracking, and financial calculations while maintaining a simple, user-friendly interface suitable for non-technical users.

---

## 2. Business Requirements

### 2.1 Primary Objectives
- Digitize client measurement records for easy retrieval and updates
- Manage multiple orders per client with detailed requirements
- Calculate order costs and generate bills
- Replace paper-based record-keeping with a searchable digital system
- Provide quick access to historical client data

### 2.2 User Persona
- **Primary User:** Home-based tailor (non-technical)
- **Technical Proficiency:** Basic computer/tablet usage
- **Usage Environment:** Home workshop, possibly with limited space
- **Frequency:** Daily usage, multiple times per day
- **Key Needs:** Quick data entry, easy search, minimal clicks

---

## 3. Functional Requirements

### 3.1 Client Management Module

#### 3.1.1 Client Registration
**Priority:** P0 (Critical)

**Features:**
- Add new client with basic information
- Required fields:
  - Full Name
  - Phone Number (primary contact)
  - Secondary Phone (optional)
  - Address (optional but recommended)
  - Date of Registration (auto-captured)
- Optional fields:
  - Email
  - Notes/Preferences
- Unique client ID generation (auto-generated)
- Duplicate detection based on phone number

**Validation Rules:**
- Phone number: 10 digits, numeric
- Name: Required, minimum 2 characters
- Prevent duplicate phone numbers with warning

#### 3.1.2 Client Search & Listing
**Priority:** P0 (Critical)

**Features:**
- Search by:
  - Name (partial match, case-insensitive)
  - Phone number
  - Client ID
- Display client list with:
  - Name
  - Phone number
  - Last order date
  - Number of active orders
- Pagination (20 clients per page)
- Sort by: Name, Last Order Date, Registration Date

#### 3.1.3 Client Profile View
**Priority:** P0 (Critical)

**Features:**
- View complete client details
- Display all measurements
- Show order history (all orders with status)
- Edit client information
- Quick action buttons:
  - Add New Order
  - Update Measurements
  - View All Orders

### 3.2 Measurement Management Module

#### 3.2.1 Measurement Recording
**Priority:** P0 (Critical)

**Measurement Categories:**

**For Shirt/Blouse:**
- Length
- Shoulder width
- Sleeve length
- Chest/Bust
- Waist
- Hip (for longer shirts)
- Armhole
- Neck
- Front neck depth
- Back neck depth
- Additional notes field

**For Churidar/Salwar:**
- Top: Same as shirt measurements
- Bottom (Churidar/Salwar):
  - Length
  - Waist
  - Hip
  - Thigh (for churidar)
  - Bottom opening (for churidar)
  - Knee (optional)
- Dupatta: Length & Width

**For Blouse:**
- Length
- Shoulder
- Sleeve length
- Bust
- Waist
- Underbust
- Armhole
- Neck (front & back)
- Back opening depth
- Princess cut measurements (if applicable)

**For Pants/Trousers:**
- Length
- Waist
- Hip
- Thigh
- Knee
- Bottom opening
- Rise (front & back)

**For Skirt:**
- Length
- Waist
- Hip
- Bottom width (for flared skirts)

**For Special Garments:**
- Customizable field list
- Template-based measurement sets

**Features:**
- Unit selection: Inches (default) or Centimeters
- Decimal precision: Up to 0.5 increments
- Measurement templates for quick entry
- Last updated timestamp
- Measurement history (version control)

#### 3.2.2 Measurement Updates
**Priority:** P1 (High)

**Features:**
- Edit existing measurements
- Copy measurements from previous version
- Compare current vs. previous measurements (side-by-side)
- Add update notes/reason for change
- Maintain measurement history (last 5 versions)
- Warning if measurements changed significantly (>2 inches)

### 3.3 Order Management Module

#### 3.3.1 Create New Order
**Priority:** P0 (Critical)

**Order Information:**
- Order ID (auto-generated: ORD-YYYYMMDD-XXXX)
- Client selection (linked to client ID)
- Order date (auto-captured)
- Expected delivery date (user-defined)
- Priority level: Normal, Urgent

**Garment Details:**
- Garment type (dropdown):
  - Shirt
  - Blouse
  - Churidar Set
  - Salwar Set
  - Pants
  - Skirt
  - Lehenga
  - Saree Blouse
  - Kids Wear
  - Custom/Other
- Quantity
- Fabric details:
  - Type (Cotton, Silk, Synthetic, etc.)
  - Client-provided or shop-provided
  - Fabric measurements (if applicable)
- Design reference:
  - Text description
  - Image upload (optional, for future)
  - Reference number (if from catalog)
- Special instructions/customization notes

**Measurement Association:**
- Use existing measurements (select measurement version)
- Enter new measurements for this order
- Quick measurement override for specific items

#### 3.3.2 Order Listing & Tracking
**Priority:** P0 (Critical)

**Features:**
- View all orders with filters:
  - Status (Pending, In Progress, Completed, Delivered, Cancelled)
  - Date range
  - Client name
  - Garment type
  - Priority
- Display columns:
  - Order ID
  - Client Name
  - Garment Type
  - Order Date
  - Delivery Date
  - Status
  - Amount
- Color coding for:
  - Overdue orders (red)
  - Due today (orange)
  - Due this week (yellow)
- Quick status update from list view
- Bulk actions: Print multiple orders, Update status

#### 3.3.3 Order Status Management
**Priority:** P1 (High)

**Status Workflow:**
1. **Order Placed** (Initial state)
   - Auto-set on order creation
   
2. **Fabric Received** (For client-provided fabric)
   - Optional status
   
3. **Cutting Done**
   - Intermediate stage
   
4. **Stitching in Progress**
   - Active work state
   
5. **Trial Pending**
   - Ready for fitting trial
   
6. **Alterations Required**
   - Post-trial status
   
7. **Completed**
   - Stitching finished
   
8. **Ready for Pickup**
   - Final state before delivery
   
9. **Delivered**
   - Order fulfilled
   
10. **Cancelled**
    - Order cancelled (with reason)

**Features:**
- Status update with timestamp
- Optional notes for each status change
- Status history log
- Automatic status-based notifications (future feature)

### 3.4 Billing & Pricing Module

#### 3.4.1 Cost Calculation
**Priority:** P0 (Critical)

**Pricing Components:**
- Base stitching charges (per garment type)
- Customization charges:
  - Embroidery work
  - Lining
  - Piping
  - Special designs
  - Additional components (buttons, hooks, etc.)
- Material charges (if provided by tailor)
- Alteration charges
- Urgent order surcharge (percentage or fixed)

**Features:**
- Predefined price templates for common garments
- Line item addition (add/remove charge items)
- Automatic subtotal calculation
- Tax/GST calculation (optional, future)
- Discount application:
  - Fixed amount
  - Percentage
  - Reason for discount
- Round-off to nearest rupee
- Display: Subtotal, Discount, Final Amount

#### 3.4.2 Payment Tracking
**Priority:** P1 (High)

**Features:**
- Advance payment recording:
  - Amount
  - Date
  - Payment method (Cash, UPI, Card, etc.)
  - Receipt number (optional)
- Balance calculation (Total - Advance)
- Final payment recording
- Payment history for each order
- Payment status indicators:
  - Not Paid
  - Partially Paid
  - Fully Paid
- Outstanding balance alerts

#### 3.4.3 Billing & Receipts
**Priority:** P1 (High)

**Features:**
- Generate itemized bill:
  - Business name/logo (configurable)
  - Order details
  - Client details
  - Itemized charges
  - Payment breakdown
  - Balance due
- Print-friendly format (A5/A6 size)
- Receipt numbering system
- PDF generation (future feature)
- SMS receipt (future feature)

### 3.5 Dashboard & Reports

#### 3.5.1 Dashboard
**Priority:** P1 (High)

**Key Metrics:**
- Orders today: New, In Progress, Completed
- Orders this week
- Orders pending delivery (grouped by date)
- Overdue orders count
- Revenue this month
- Outstanding payments total

**Quick Actions:**
- Add New Client
- Create New Order
- Search Client
- View Today's Deliveries

**Widgets:**
- Recent orders (last 10)
- Upcoming deliveries (next 7 days)
- Clients with pending orders
- Payment reminders

#### 3.5.2 Reports Module
**Priority:** P2 (Medium)

**Reports:**
1. **Order Report:**
   - Date range selection
   - Filter by status, client, garment type
   - Export as CSV/PDF
   - Metrics: Total orders, completed, pending, cancelled

2. **Revenue Report:**
   - Daily/Weekly/Monthly/Custom date range
   - Total revenue, advances, balance collected
   - Payment method breakdown
   - Garment type revenue distribution

3. **Client Report:**
   - Most frequent clients
   - Client lifetime value
   - New clients in period

4. **Performance Report:**
   - Average completion time by garment type
   - On-time delivery percentage
   - Orders per day/week/month trend

## 4. Non-Functional Requirements

### 4.1 Performance
- Page load time: < 2 seconds
- Search results: < 1 second
- Support up to 1000 clients, 5000 orders without degradation
- Smooth operation on mid-range devices (tablets/laptops)

### 4.2 Usability
- Intuitive UI requiring minimal training
- Large, touch-friendly buttons for tablet use
- Clear visual hierarchy
- Minimal clicks to complete tasks (max 3 clicks for common tasks)
- Error messages in simple, actionable language
- Confirmation dialogs for destructive actions

### 4.3 Reliability
- Auto-save for data entry forms
- Data backup mechanism (export/import)
- Graceful error handling
- No data loss on browser refresh/crash

### 4.4 Security
- Local authentication (password/PIN protection)
- Data encryption for sensitive information
- Session timeout for inactive users
- Role-based access (future: for assistants/staff)

### 4.5 Compatibility
- Modern browsers: Chrome, Firefox, Safari, Edge (latest 2 versions)
- Responsive design: Desktop (1366x768+), Tablet (iPad size), Mobile (fallback)
- Offline capability (future enhancement)

### 4.6 Maintainability
- Clean, documented code
- Modular architecture for easy updates
- Simple deployment process
- Easy backup and restore procedures

## 5. Technical Architecture

### 5.1 Technology Stack

#### Frontend
- **Framework:** React 18+
- **UI Library:** Tailwind CSS + shadcn/ui components
- **State Management:** React Context API / Zustand (lightweight)
- **Form Handling:** React Hook Form
- **Validation:** Zod
- **Date Handling:** date-fns
- **Icons:** Lucide React
- **Routing:** React Router v6

#### Backend/Database
- **Local Storage:** IndexedDB (via Dexie.js)
  - Structured database with indexing
  - Supports complex queries
  - Good performance for large datasets
  - Offline-first capability
  
#### Alternative Options for Future:
- **Phase 2 (Cloud Migration):**
  - Backend: Node.js + Express / Python + FastAPI
  - Database: PostgreSQL / MongoDB
  - Hosting: Vercel/Netlify (frontend) + Railway/Render (backend)
  - Authentication: JWT tokens
  - API: RESTful or GraphQL

### 5.2 Database Schema Design

#### Collections/Tables:

**1. Clients Table**
```javascript
{
  id: "CLI-20251102-0001", // Primary Key
  name: String, // Required
  phoneNumber: String, // Required, Unique
  secondaryPhone: String, // Optional
  address: String,
  email: String,
  notes: String,
  registrationDate: Date,
  lastOrderDate: Date, // Updated automatically
  totalOrders: Number, // Counter
  createdAt: Timestamp,
  updatedAt: Timestamp
}
```

**2. Measurements Table**
```javascript
{
  id: "MEAS-UUID", // Primary Key
  clientId: String, // Foreign Key -> Clients
  garmentType: String, // shirt, blouse, churidar, etc.
  measurements: {
    // Dynamic based on garment type
    length: Number,
    shoulder: Number,
    sleeve: Number,
    chest: Number,
    waist: Number,
    hip: Number,
    // ... other measurements
  },
  unit: String, // "inches" or "cm"
  notes: String,
  version: Number, // Measurement version
  isActive: Boolean, // Latest version flag
  createdAt: Timestamp,
  createdBy: String
}
```

**3. Orders Table**
```javascript
{
  id: "ORD-20251102-0001", // Primary Key
  clientId: String, // Foreign Key -> Clients
  clientName: String, // Denormalized for quick access
  clientPhone: String, // Denormalized
  
  orderDate: Date,
  deliveryDate: Date,
  priority: String, // "normal" or "urgent"
  
  garmentType: String,
  quantity: Number,
  
  fabricDetails: {
    type: String,
    providedBy: String, // "client" or "tailor"
    measurements: String,
    notes: String
  },
  
  designDetails: {
    description: String,
    referenceNumber: String,
    imageUrl: String // For future
  },
  
  measurementId: String, // Foreign Key -> Measurements
  measurementSnapshot: Object, // Copy of measurements at order time
  
  specialInstructions: String,
  
  status: String, // Enum: placed, fabric_received, cutting, stitching, trial, alterations, completed, ready, delivered, cancelled
  statusHistory: [
    {
      status: String,
      timestamp: Date,
      notes: String
    }
  ],
  
  pricing: {
    baseCharge: Number,
    customizations: [
      {
        description: String,
        amount: Number
      }
    ],
    materialCharges: Number,
    urgentCharges: Number,
    subtotal: Number,
    discount: {
      amount: Number,
      reason: String
    },
    total: Number
  },
  
  payments: [
    {
      amount: Number,
      date: Date,
      method: String, // cash, upi, card, etc.
      type: String, // "advance" or "final"
      receiptNumber: String,
      notes: String
    }
  ],
  
  totalPaid: Number,
  balanceDue: Number,
  paymentStatus: String, // "not_paid", "partially_paid", "fully_paid"
  
  createdAt: Timestamp,
  updatedAt: Timestamp,
  completedAt: Timestamp,
  deliveredAt: Timestamp
}
```

**4. Garment Templates Table**
```javascript
{
  id: String, // Primary Key
  name: String, // e.g., "Standard Shirt"
  garmentType: String,
  measurementFields: [
    {
      fieldName: String,
      label: String,
      required: Boolean,
      defaultValue: Number
    }
  ],
  defaultPrice: Number,
  createdAt: Timestamp
}
```

**5. Settings Table (Key-Value)**
```javascript
{
  key: String, // Primary Key
  value: Any,
  updatedAt: Timestamp
}
// Examples:
// businessName, businessAddress, phoneNumber, logo, gstNumber, defaultUnit, etc.
```

### 5.5 Key Technical Decisions

#### Why IndexedDB (via Dexie.js)?
1. **Structured Data:** Supports complex queries and indexes
2. **Performance:** Much faster than localStorage for large datasets
3. **Storage Capacity:** Can store several hundred MB (vs 5-10MB for localStorage)
4. **Offline-First:** Works completely offline
5. **Transaction Support:** ACID compliance for data integrity
6. **Easy Migration Path:** Can sync to cloud backend later

#### Why React?
1. **Component Reusability:** Build once, use everywhere
2. **Large Ecosystem:** Many libraries and tools available
3. **Easy to Learn:** Good documentation and community
4. **Performance:** Virtual DOM for efficient updates
5. **Mobile Ready:** Can convert to React Native later

#### Why Tailwind + shadcn/ui?
1. **Rapid Development:** Pre-built components
2. **Consistent Design:** Design system out of the box
3. **Responsive:** Mobile-first approach
4. **Customizable:** Easy to adapt to brand colors
5. **Accessibility:** Built-in a11y features
